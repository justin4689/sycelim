## News

- **Eyebrow:** News & perspectives
- **Title:** What we think about building software that lasts.
- **Lead:** Engineering notes, methodology and field lessons from the systems we build and run.

### Article 1 - Why we call ourselves an engineering company, not a web agency
*Category: Perspective · Date: 5 June 2026*

**Excerpt:** The difference is not just wording: it changes how software is designed, built and maintained over time.

Many software projects fail not for lack of talent, but for lack of method. Code comes before understanding the problem, features pile up without architecture, and technical debt eventually blocks every evolution.

**Design before code**
At Sycelim, a line of production code is written only after deliberately deciding the architecture, the data model and the trade-offs. That is where a system's scalability and longevity are decided.

**What it changes in practice**
- Decisions documented and traceable, not made on the fly.
- Industrialised quality: code review, automated testing, continuous integration.
- Security and compliance built in from the design, not bolted on afterwards.

> A poorly engineered system costs more than it returns.

This discipline is what lets us deploy a single business engine across several organisations, beyond borders, without rebuilding everything each time.

### Article 2 - Security is not bolted on at the end: it is designed from day one
*Category: Security · Date: 22 May 2026*

**Excerpt:** Treating security as a final step means you have already lost. In a system that handles sensitive data, protection is an architectural decision.

When a system handles memberships, contributions or health data, security is not one feature among others: it is a condition of existence. Yet it is still too often pushed back to a last-minute audit, right before going live.

**The cost of late security**
Fixing a flaw after the fact forces you to revisit the architecture, the data model and sometimes the user experience. What could have been a design decision becomes a costly, high-pressure rework - with the risk already exposed.

**Our approach: protection by construction**
- Access control and role separation decided at the architecture level, not in the interface.
- Encryption of sensitive data and protection of exchanges between services and partners.
- APIs designed to be exposed securely: authentication, validation, traceability.
- Regulatory compliance built into the business model, not laid over it.

> Security is not what slows a project down: it is what lets it last.

Designing a system that is compliant by construction - auditable, defensible, ready to follow regulation - costs less than securing it after an incident. That is why, with us, a cybersecurity engineer is involved from the very first design decisions, not at the end.

### Article 3 - Technical debt: the hidden cost that eventually slows everything down
*Category: Method · Date: 8 May 2026*

**Excerpt:** Software that becomes slower and slower to evolve does not have a speed problem: it has a debt problem. Here is how we avoid it - and how to pay it down.

Technical debt is everything you postpone to ship faster: an architectural shortcut, a test you skip, documentation you defer. Like financial debt, it is repaid with interest - and that interest is paid on every future change.

**How to recognise it**
- Every new feature takes longer than the previous one.
- Nobody dares touch certain parts of the code that "just work".
- Bug fixes create new bugs elsewhere.

**Avoid it rather than endure it**
The best technical debt is the one you never take on. Architecture thought through upfront, systematic code review, automated tests and continuous integration are not wasted time: they are the guardrails that keep a system flexible as it grows.

**Modernise without breaking everything**
When the debt is already there, the answer is almost never to rewrite everything at once. We map the existing system, isolate the risk areas, secure them with tests, then modernise incrementally - keeping the system running at every step.

> A system you no longer dare to change has stopped being an asset and become a risk.
